# OmniSyntax QA Report

Date: 2026-03-27

## Executive Summary

OmniSyntax was tested across `C`, `Java`, and `JavaScript` using 21 targeted cases covering syntax errors, logical errors, runtime errors, and edge cases.

Baseline findings before fixes:

- `C`: 4 hard failure cases out of 7. The system missed division by zero, missing include, undeclared identifier, and type mismatch.
- `Java`: 3 hard failure cases out of 7, plus 1 partial detection. The system missed division by zero, missing import, undeclared identifier, and only partially handled the provided mixed-error sample.
- `JavaScript`: 3 hard failure cases out of 7, plus 1 partial detection. The system missed undefined identifiers, unreachable code, duplicate definitions, and only partially handled the provided mixed-error sample.
- Root cause: the ML layer was degraded in this environment because the serialized scikit-learn model bundle is incompatible with the installed runtime. That left C/Java/JavaScript heavily dependent on shallow rule checks.

Post-fix status:

- The rule engine now detects common C/Java/JavaScript semantic and runtime issues even when ML is unavailable.
- C/Java/JavaScript rule-based findings now include exact line and column metadata.
- Multi-error reporting now returns multiple findings for non-Python languages as well.
- Cascading bracket noise after unclosed strings is now suppressed so tutoring stays focused on the primary fix.
- Validation: `python -m pytest tests -q` => `87 passed, 1 xfailed`.

## Root Cause Analysis

### 1. ML dependency failure

The biggest systemic issue was environmental:

- The primary serialized model bundle was trained against an older scikit-learn version.
- The current environment could partially deserialize metadata but failed to load the full estimator graph.
- The fallback legacy model files were incomplete in this checkout.

Effect:

- Semantic classifications silently disappeared unless a rule-based fallback existed.
- C/Java/JavaScript suffered the most because their structural path previously had very limited semantic coverage.

### 2. C/Java/JavaScript structural path was too shallow

Before the fix, non-Python detection mostly reduced to:

- bracket balance
- missing semicolon
- unclosed string
- optional ML classification

Effect:

- obvious classroom errors such as `int x = "10";`, `ArrayList` without import, `console.log(y);`, and `10 / 0` were missed.

### 3. Missing location data blocked useful tutoring

Pre-fix non-Python results did not expose line numbers for most findings.

Effect:

- tutor explanations were generic
- auto-fix had limited targeting context
- bug reports were harder to reproduce

## Architecture Recommendations

1. Replace pickle-coupled ML deployment with versioned model artifacts.
   - Store training-library metadata with the bundle.
   - Fail fast during startup when model/runtime versions diverge.
   - Prefer a portable inference format or retrain with a supported dependency set.

2. Add parser-grade frontends per language.
   - `C/C++`: `libclang` or tree-sitter
   - `Java`: tree-sitter Java or compiler-front parser
   - `JavaScript`: `acorn`, `esprima`, or tree-sitter

3. Separate result sources.
   - Return whether a finding came from `parser`, `rule`, or `ml`.
   - This makes QA and pedagogy easier and prevents hidden coupling.

4. Preserve multi-error output everywhere.
   - The tutoring UX should expose a primary error plus secondary findings in severity order.

5. Add CI accuracy gates for curated regressions.
   - Keep a fixed multi-language corpus of syntax, runtime, logical, and negative cases.
   - Track false negatives and false positives separately.

6. Improve malformed-input suppression.
   - When an unclosed string exists, suppress follow-on bracket noise from the same line unless confidence is high.

## Final Assessment

OmniSyntax is now materially stronger for offline or degraded execution. The largest remaining production blocker is the ML deployment issue, not the rule engine. If the project is intended for programming education, restoring a reliable model bundle and introducing parser-backed language modules should be the next priority after this patch set.
