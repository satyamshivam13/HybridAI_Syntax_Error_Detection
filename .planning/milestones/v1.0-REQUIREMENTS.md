# Requirements Archive: OmniSyntax v1.0

**Archived:** 2026-03-27
**Milestone:** v1.0 — Cross-Language Reliability
**Core Value:** Students should get accurate, actionable code-error feedback even when the ML layer is unavailable.

---

## v1 Requirements — ALL COMPLETE ✅

### Detection

- [x] **DET-01**: User can submit Python, Java, C, C++, or JavaScript code and receive a primary analysis result.
  *Outcome: All 5 languages supported through API, CLI, and Streamlit with ML + rule-based detection.*

- [x] **DET-02**: User receives useful C, Java, and JavaScript issue detection even when ML models are unavailable.
  *Outcome: 11+ rule-based patterns added for C/Java/JavaScript; degraded mode confirmed to produce actionable output.*

- [x] **DET-03**: User can see multiple issues for a snippet when the code contains more than one detectable problem.
  *Outcome: `--all-errors` CLI flag and `/check` multi-error path both enumerate all findings.*

- [x] **DET-04**: User does not receive bracket or delimiter false positives caused only by characters inside string literals.
  *Outcome: String-literal stripping added to rule checks; regression tests confirm false positives are suppressed.*

### Diagnostics

- [x] **DIAG-01**: User receives exact line information for detected errors whenever the engine can localize the issue.
  *Outcome: Line numbers populated in all rule-based issue objects for Python, C, C++, Java, and JavaScript.*

- [x] **DIAG-02**: User receives column information for rule-based C, Java, and JavaScript findings when available.
  *Outcome: Column offsets added to rule-based findings in `_collect_c_like_rule_based_issues`.*

- [x] **DIAG-03**: User receives an explanation that states what is wrong and what to fix next.
  *Outcome: `tutor_explainer.py` updated; non-generic explanations asserted in regression tests.*

- [x] **DIAG-04**: User sees degraded-mode warnings when semantic ML analysis is unavailable.
  *Outcome: Warnings propagated through `multi_error_detector.py` and rendered in CLI and Streamlit.*

### Autofix and UX

- [x] **FIX-01**: User can request an autofix for supported syntax issues without unrelated code mutation.
  *Outcome: Auto-fix returns fixed code for MissingDelimiter, MissingInclude, MissingImport, MissingColon, UnclosedString.*

- [x] **FIX-02**: User gets consistent issue labels and wording across API, CLI, and Streamlit entry points.
  *Outcome: Shared label set from `error_engine.py`; parity tested in Phase 05-04 sign-off.*

- [x] **FIX-03**: User can override language selection when filename-based detection would be misleading.
  *Outcome: `language` override supported in `POST /check` and `POST /check-and-fix`.*

### Reliability and Operations

- [x] **OPS-01**: API `/health` accurately reports healthy versus degraded runtime state.
  *Outcome: Health contract tested in `test_api_and_regressions.py`; 18 passing health/regression tests.*

- [x] **OPS-02**: API payload limits and rate limits are enforced consistently across endpoints.
  *Outcome: Payload limit enforced across `/check`, `/fix`, `/quality` via shared middleware.*

- [x] **OPS-03**: CLI, API, and Streamlit startup paths work on a clean local environment with documented setup.
  *Outcome: Startup instructions in README; smoke suite validates all three entry points.*

### QA and Evaluation

- [x] **QA-01**: Automated tests cover known false negatives and regressions for C, Java, and JavaScript.
  *Outcome: `test_c_java_js_regressions.py` — 10 regression tests; `test_cross_language_error_coverage.py` — 20 cases.*

- [x] **QA-02**: Smoke scripts fail gracefully when ML artifacts are unavailable instead of crashing.
  *Outcome: `test_script_smoke.py` — 13 tests; graceful degradation signal verified.*

- [x] **QA-03**: Documentation and reports clearly describe current limitations, fixes, and verification evidence.
  *Outcome: `docs/QA_REPORT_2026-03-27.md` updated post-Phase-05; README documents healthy/degraded expectations.*
